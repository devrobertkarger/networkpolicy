<p dir="auto">NetworkPolicy is a standardized Kubernetes object to control the allowed network traffic patterns between Kubernetes pods and namespaces as well as any traffic entering or leaving the cluster.
However, Kubernetes itself does not provide an implementation of NetworkPolicy, it is typically provided by the <a href="https://kubernetes.io/docs/concepts/extend-kubernetes/compute-storage-net/network-plugins/" rel="nofollow">CNI plugin</a>.
If no NetworkPolicy is loaded, all communication is allowed which is clearly violating least-privilege and zero-trust security privileges.
For more information on the concept, see <a href="https://kubernetes.io/docs/concepts/services-networking/network-policies/" rel="nofollow">Network Policies</a> in the Kubernetes documentation.</p>

<h1 dir="auto"><a id="user-content-fundamentals" class="anchor" aria-hidden="true" href="#fundamentals"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Fundamentals</h1>
<p dir="auto">NetworkPolicy is a standardized Kubernetes object to control the allowed network traffic patterns between Kubernetes pods and namespaces as well as any traffic in and out of a cluster.
However, Kubernetes itself does not provide an implementation of NetworkPolicy, it is typically provided by the <a href="https://kubernetes.io/docs/concepts/extend-kubernetes/compute-storage-net/network-plugins/" rel="nofollow">CNI plugin</a>.
If no NetworkPolicy is loaded, all communication is allowed which is clearly violating least-privilege and zero-trust security privileges.
It is thus in the best interest of any Kubernetes cluster operator to understand and use the concept of NetworkPolicy.</p>
<p dir="auto"><a target="_blank" rel="noopener noreferrer" href="/networkpolicy/tutorial/blob/master/images/networkpolicy.png"><img src="/networkpolicy/tutorial/raw/master/images/networkpolicy.png" alt="" style="max-width: 100%;"></a></p>
<p dir="auto">When referring to NetworkPolicy more broadly, users may refer to both the standard Kubernetes NetworkPolicy as well as the extended version with additional capabilities specific to the selected <a href="https://kubernetes.io/docs/concepts/extend-kubernetes/compute-storage-net/network-plugins/" rel="nofollow">CNI plugin</a>, e.g. <a href="https://docs.cilium.io/en/v1.8/concepts/kubernetes/policy/#ciliumnetworkpolicy" rel="nofollow">CiliumNetworkPolicy</a>.
These extended versions are typically implemented as <a href="https://kubernetes.io/docs/concepts/extend-kubernetes/api-extension/custom-resources/" rel="nofollow">Custom Resources</a>.
In the initial chapters of this tutorial, we will focus on Kubernetes NetworkPolicy and then dive into some of the extensions to understand benefits and tradeoffs.</p>
<h3 dir="auto"><a id="user-content-101---namespaces-pods--services" class="anchor" aria-hidden="true" href="#101---namespaces-pods--services"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>101 - Namespaces, Pods, &amp; Services</h3>
<p dir="auto">In order to understand NetworkPolicy, you need to understand the basics of Kubernetes objects.
This is a quick 101 of the Kubernetes objects that are relevant for NetworkPolicy.
If you are already familiar with Kubernetes objects, feel free to skip this section.
If are looking for a more detailed introduced, refer to the Kubernetes documentation <a href="https://kubernetes.io/docs/concepts/overview/working-with-objects/" rel="nofollow">Working with Kubernetes Objects</a>.</p>
<p dir="auto"><a target="_blank" rel="noopener noreferrer" href="/networkpolicy/tutorial/blob/master/images/resources.png"><img src="/networkpolicy/tutorial/raw/master/images/resources.png" alt="" style="max-width: 100%;"></a></p>
<h3 dir="auto"><a id="user-content-namespaces" class="anchor" aria-hidden="true" href="#namespaces"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a><a href="https://kubernetes.io/docs/concepts/overview/working-with-objects/namespaces/" rel="nofollow">Namespaces</a></h3>
<p dir="auto">A namespace groups Kubernetes objects into a specific context.
If you want, you can consider it a virtual cluster.
Namespaces allow to share a cluster among multiple tenants.
In the context of NetworkPolicy, it is most common to define the scope of a NetworkPolicy to only apply within a particular namespace and specifically handle any cross-namespace communication.
More on this later.</p>
<h3 dir="auto"><a id="user-content-pods" class="anchor" aria-hidden="true" href="#pods"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a><a href="https://kubernetes.io/docs/concepts/workloads/pods/" rel="nofollow">Pods</a></h3>
<p dir="auto">Pods are the smallest deployable unit of an application that you can create and manage. Pods are typically created via a higher-level object such as a <a href="https://kubernetes.io/docs/concepts/workloads/controllers/deployment/" rel="nofollow">Deployments</a>, <a href="https://kubernetes.io/docs/concepts/workloads/controllers/daemonset/" rel="nofollow">DaemonSets</a>, <a href="https://kubernetes.io/docs/concepts/workloads/controllers/job/" rel="nofollow">Jobs</a>, <a href="https://kubernetes.io/docs/concepts/workloads/controllers/cron-jobs/" rel="nofollow">StatefulSets</a>, etc. and are deployed directly into a particular <a href="https://kubernetes.io/docs/concepts/overview/working-with-objects/namespaces/" rel="nofollow">Namespace</a>. A pod consist of one or more application containers which all share the same storage and network resources and always run on the same host. If your application is scalable and deployed with multiple replicas then each replica is deployed as an individual pod.
Pods can be assigned a set of <a href="https://kubernetes.io/docs/concepts/overview/working-with-objects/labels/" rel="nofollow">Labels</a>. These labels allow to select any number of pods and treat them in artbirary groups using <a href="https://kubernetes.io/docs/concepts/overview/working-with-objects/labels/#label-selectors" rel="nofollow">Selectors</a>. In the context of NetworkPolicy, a policy is applied to one or more pods by selecting pod labels in the NetworkPolicy. The policy is then applied to all application containers of any pod that matches the label selector.</p>
<h3 dir="auto"><a id="user-content-services" class="anchor" aria-hidden="true" href="#services"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a><a href="https://kubernetes.io/docs/concepts/services-networking/service/" rel="nofollow">Services</a></h3>
<p dir="auto">Applications running as pods are made available to other pods and to external clients using services. A service makes a set of pods available using an internal or external IP address and assigns a stable DNS name to this IP to allow for the discovery of the service by other pods. In the context of NetworkPolicy, services typically play a minor role as policies are applied to pods directly but use of services can influence how NetworkPolicy has to be defined.</p>
<p dir="auto">If you want to deepen your understanding of Kubernets objects before continuing, feel free to dive into the <a href="https://kubernetes.io/docs/concepts/" rel="nofollow">Kubernetes Concepts</a> chapters of the Kubernetes documentation.</p>
<h3 dir="auto"><a id="user-content-default-allow--default-deny" class="anchor" aria-hidden="true" href="#default-allow--default-deny"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Default Allow &amp; Default Deny</h3>
<p dir="auto">In the absence of any NetworkPolicy, the default behavior of a Kubernetes
cluster is to allow everything on the network. It is your responsibility as an
operator of a Kubernetes cluster to create NetworkPolicy objects to change
the <strong>default allow</strong> setting to <strong>default deny</strong>  and the explicitely allow
all required traffic paterns.</p>
<p dir="auto">If your Kubernetes cluster is already running many applications, this may appear difficult as changing from a <strong>default allow</strong> to a <strong>default deny</strong> security model in a single step may appear challenging. Fortunately, NetworkPolicy allows to change the <strong>default deny</strong> setting for individual groups of pods or individual namespaces so you can start locking down the most important applications first and improve your security posture step by step.</p>
<p dir="auto">Before we now create our first NetworkPolicy, we need to understand one more fundamental concept:</p>
<h3 dir="auto"><a id="user-content-ingress--egress" class="anchor" aria-hidden="true" href="#ingress--egress"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Ingress &amp; Egress</h3>
<p dir="auto">Ingress and egress are terms used in networking to describe traffic direction. Ingress refers to network traffic that is transmitted into a Kubernetes pod and egress is refers to network traffic that is emmited by a pod.</p>
<p dir="auto"><a target="_blank" rel="noopener noreferrer" href="/networkpolicy/tutorial/blob/master/images/ingress_egress.png"><img src="/networkpolicy/tutorial/raw/master/images/ingress_egress.png" alt="" style="max-width: 100%;"></a></p>
<p dir="auto">The <strong>default deny</strong> setting of a pod can be control for each traffic direction separately, i.e., you can initially deny all traffic into a pod and start allowing individual traffic patterns while still allowing all network traffic leaving a pod.</p>

<p dir="auto">In the previous chapter, we put the ingress direction of the<code>backend</code> pod into
<strong>default deny</strong>. How was it possible for kubernetes.io to still be reachable
from within that pod. Shouldn't the reply packets of the network connection to
kubernetes.io have been dropped?</p>
<p dir="auto">The reason why the example worked is because NetworkPolicy is typically
enforced in a stateful manner. This is not required by the NetworkPolicy
specification and will depend on the selected <a href="https://kubernetes.io/docs/concepts/extend-kubernetes/compute-storage-net/network-plugins/" rel="nofollow">CNI plugin</a> but stateful
enforcement is the norm and typically expected by users.</p>
<p dir="auto"><a target="_blank" rel="noopener noreferrer" href="/networkpolicy/tutorial/blob/master/images/stateful.png"><img src="/networkpolicy/tutorial/raw/master/images/stateful.png" alt="" style="max-width: 100%;"></a></p>
<p dir="auto">Stateful enforcement enables allowing a network connection in the forward direction only and let the stateful enforcement automatically allow reply network traffic for known network connections. It is called stateful is because the <a href="https://kubernetes.io/docs/concepts/extend-kubernetes/compute-storage-net/network-plugins/" rel="nofollow">CNI plugin</a> will maintain state to keep track of all active network connections and associate all network packets to a connection in either the forward or reply direction.</p>

<p dir="auto">This highlights another key aspect of NetworkPolicy: Policies are additive. If you create multiple policies that select the same pod then all rules specified in all policies are added together and will apply to the pod. In our example, the <code>backend-ingress-deny</code> policy puts the pod into ingress <strong>default deny</strong> mode and the <code>backend-ingress-allow-frontend</code> allows network communication from <code>frontend</code>. Technically, the <code>backend-ingress-deny</code> policy is no longer needed as the <code>backend-ingress-allow-frontend</code> policy also puts the pod into <strong>default deny</strong>. However, you will find that using a separate policy for <strong>default deny</strong> is preferred as it avoids accidental exposure of a pod on the removal of the allow policy. It also enables you to put all pods of a namespace into <strong>default deny</strong> and then describe allow rules for each deployment in a separte policy.</p>
